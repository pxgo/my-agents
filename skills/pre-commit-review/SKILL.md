# 提交前审查与提交

在改动进入 git 历史前捕获问题；无阻塞问题且经用户确认后，自动按模块或功能分多步完成 commit。

## 核心原则

- 有阻塞问题绝不提交，只报告问题并建议修复，不请求确认。
- 始终按模块或功能拆分提交，每步内聚、步骤间不重叠。
- 提交必须经用户确认（弹窗），确认后自动逐步执行全部 commit。
- 保留必要的逻辑推理（交叉验证、根因判断、影响分析不可省），砍掉一切口水话与重复；同一信息只说一次，后续仅引用，不再展开。目的是节约 token、减小上下文。

## 工作流

### 1. 收集 diff

```sh
git --no-pager status
git --no-pager diff --stat
git --no-pager diff --staged
git --no-pager diff
```

若同时存在暂存和未暂存改动，默认只审查**暂存**部分；用户明确要求时再审查两者并分开报告。大 diff 可先 `git --no-pager diff --staged --name-only` 逐文件审查。

### 2. 加载项目约定

读取根目录及改动目录下的 `AGENTS.md`，必要时读取 `README.md`。项目规则优先级高于下方通用检查项。

### 3. 审查清单

只报告真实存在的问题，不要编造。

#### 正确性 & bug
- 逻辑错误：off-by-one、条件反转、运算符错误、变量误用（能编译但行为错）。
- 异步问题：null/undefined 访问、遗漏 `await`、未处理的 promise rejection。
- 资源泄漏：未关闭的文件、连接、监听器、定时器。
- 竞态与执行顺序问题、错误处理不当（吞异常、空 `catch`、未向上传递）。
- 边界情况：空数组、0、负数、超大输入、unicode。

#### 安全
- 硬编码的密钥、token、密码、内部 URL；误提交的 `.env`、凭据、配置文件。
- 注入风险：SQL/NoSQL 注入、命令注入、路径穿越、未转义输入导致 XSS。
- 权限校验被削弱或移除、CORS 放宽。
- 新增外部依赖 -- 提示用户确认可信度。

#### 调试残留
- **调试打印（必须作为 🟡 警告标记）：** 非测试代码中的调试打印语句（如 `console.log`、`print`、`dump` 等无用输出），不可遗漏。
- 无解释的注释代码块；本次 diff 新增的 `TODO` / `FIXME`（提示，不必阻塞）。
- 临时值：`localhost`、`127.0.0.1`、假邮箱、`test123`。
- 测试标记：`.only` / `.skip` / `fdescribe` / `xit`；`debugger` 语句。

#### 项目约定（依据 AGENTS.md）
检查文件位置、分层违规（如路由层直接访问 `ctx.db` 而未走 `services`）、文件类型（如要求 `.ts` 却新增 `.js`）、导出方式、模型约定、路径别名（`../../../` 而非 `@/`）、组件规范（如 `setup + ts + pug`）、路由同步（如 `operations.ts`）等。

#### 代码质量
- 死代码、未使用的 import/变量、不可达分支；重复逻辑建议复用已有实现。
- 过于复杂的函数、缺少命名常量的魔法数/字符串、命名风格不一致。
- 公共 API 改动未同步调用方（快速 `grep` 旧符号）。
- 注释与代码不一致：代码改了但相关注释未同步更新，作为 🟡 警告标记。

#### 类型 & API（静态类型语言）
- 无理由引入 `any` / `unknown`；`@ts-ignore` / `@ts-expect-error` / `eslint-disable` 必须有说明。
- 用类型断言 `as Foo` 掩盖真实类型不匹配；公共函数签名改动未同步 `types/` 定义。

#### 测试
- 项目有测试套件时新行为却无测试；测试没有实质断言；快照更新疑似掩盖真实回归。

#### 文档 & i18n
- 公共 API / 配置 / CLI 选项变更未更新文档；用户可见文案硬编码未走 i18n（如 `languages/`）。

#### diff 卫生
- 无关改动混入建议拆分；大段格式化混入逻辑改动建议分开提交。
- 大型生成文件（lockfile、构建产物）确认是否应入库；不应追踪的文件：`.DS_Store`、`Thumbs.db`、IDE 配置、`node_modules`、`dist`、`.env*`。
- 仅行尾/空白差异的噪声；遗留冲突标记：`<<<<<<<`、`=======`、`>>>>>>>`。

### 4. 交叉验证

改动可疑时**先核实再报告**：函数签名变更 -> `grep` 调用方；文件删除/重命名 -> `grep` 旧路径；配置键重命名 -> 搜索旧键。

### 5. 制定提交策略

按**模块或功能**拆分提交步骤，始终分多步提交（单一模块单一主题时 N=1，属特例）。

- 按模块/领域或功能主题拆分（如功能新增一步、bug 修复一步、依赖升级一步）。
- 大段格式化/重排与逻辑改动分开；新文件入库与无关杂项改动分开。
- 步骤顺序符合依赖关系（如先底层后上层、先重构后功能）。
- 每步可独立提交、独立回滚，文件清单明确且步骤间不重叠。

### 6. 输出审查结果与提交方案

全文使用**中文**（文件路径、行号、代码片段、命令、英文术语保持原样），结构如下：

```
## 提交前审查

**范围：** 已暂存 N 个文件，未暂存 M 个文件

### 🔴 阻塞问题
- `path/to/file.ts:42` -- <问题及原因>
  建议修复：<方案>

### 🟡 警告
- `path/to/file.ts:88` -- <需要再确认的点>

### 🟢 提示
- ...

### 总结
<一句话："可以提交" 或 "请先修复阻塞问题后再次审查">

### 提交方案
**步骤 1 / N** -- <模块/主题说明>
信息：`<type>: <描述>`
文件：`path/a.ts`, `path/b.ts`

**步骤 2 / N** -- <模块/主题说明>
信息：`<type>: <描述>`
文件：`path/c.ts`
```

**输出规则：**
- 始终使用 diff 新版本侧的行号。
- 无阻塞问题时总结为"可以提交"，输出方案后通过 `ask-user-choice` 弹窗请求确认；有阻塞问题时总结为"请先修复阻塞问题后再次审查"，**不输出提交方案，不请求确认**。
- 不写填充语，不建议与本次 diff 无关的重构。

### 7. 请求确认并执行 commit

仅当**无阻塞问题**时，调用 `ask-user-choice` skill 弹窗请求确认（不在聊天框等待手动输入）。

**弹窗问题文本须自包含完整信息**（用户仅看弹窗即可理解，无需翻看聊天记录），包含：审查结果摘要（阻塞/警告/提示数量，若有警告或提示逐条简述）、提交方案详情（总步骤数及每步 commit message 和文件清单）、确认提示语。

用户确认 -> 自动逐步执行全部 commit；取消 -> 不执行任何 git 操作并告知用户；用户选"其他"并输入自定义内容 -> 按用户意图处理（如要求修改方案则调整后重新弹窗确认）。

**执行 commit：** 按方案步骤顺序，每步仅暂存该步文件（`git add -- <files>`）并提交（`git commit -m "<type>: <描述>"`）；逐步自动执行无需逐步确认；全部完成后 `git --no-pager log -n <步数> --stat` 汇总反馈。严格按文件清单暂存不夹带其他改动，commit 信息与方案一致不加 body。执行中发现某步有阻塞问题则停止该步及后续步骤并提示用户。

## commit message 规则

- 格式：`<type>: <描述>`，不带 body，整行 50 字符左右。
- type：`feat`（新增功能/接口/页面）、`fix`（修复 bug）、`perf`（性能优化）、`ref`（重构）、`chore`（杂项：构建、依赖、脚本等）。
- 描述以动词开头，不以标点结尾。示例：`feat: 新增专栏置顶接口及权限校验` / `fix: 修复登录后 Token 未刷新`。

## 不适用场景

- 工作区干净 -- 告知无内容可审查。
- 整个项目架构审查 -- 属于另一类任务。
