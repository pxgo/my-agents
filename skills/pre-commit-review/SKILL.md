# 提交前审查与提交

在改动进入 git 历史前捕获问题；无阻塞问题且经用户确认后，自动按模块或功能分多步完成 commit。

## 核心原则

- **有阻塞问题绝不提交**：存在 🔴 阻塞问题时，只报告问题并建议修复，不进入提交流程，也不请求确认。
- **始终按模块或功能拆分提交**：改动按模块/功能主题拆分为多步 commit，每步内聚、步骤间不重叠。拆分与执行全程自动化，用户只需在弹窗中确认或取消。
- **提交必须经用户确认**：无阻塞问题时，先展示 commit 方案，通过 `ask-user-choice` 弹窗请求用户确认。用户确认后自动逐步执行全部 commit，无需人工干预；用户取消则不执行。

## 工作流

### 1. 收集 diff

```sh
git --no-pager status
git --no-pager diff --stat
git --no-pager diff --staged
git --no-pager diff
```

若同时存在暂存和未暂存改动，默认只审查**暂存**部分；用户明确要求时再审查两者并分开报告。大 diff 可先 `git --no-pager diff --staged --name-only` 逐文件审查。

### 2. 加载项目约定

- 读取根目录及改动目录下的 `AGENTS.md`。
- 必要时读取 `README.md`。
- 项目规则优先级高于下方通用检查项。

### 3. 审查清单

只报告真实存在的问题，不要为了显得全面而编造。

#### 正确性 & bug
- 逻辑错误、off-by-one、条件反转、运算符错误。
- null/undefined 访问、遗漏 `await`、未处理的 promise rejection。
- 变量误用（能编译但行为错）。
- 资源泄漏：未关闭的文件、连接、监听器、定时器。
- 竞态与执行顺序问题。
- 错误处理：吞异常、空 `catch`、未向上传递错误。
- 边界情况：空数组、0、负数、超大输入、unicode。

#### 安全
- 硬编码的密钥、token、密码、内部 URL。
- 误提交的 `.env`、凭据、配置文件。
- SQL/NoSQL 注入、命令注入、路径穿越。
- 未转义的用户输入渲染到 HTML（XSS）或拼接到 shell。
- 权限校验被削弱或移除、CORS 放宽。
- 新增外部依赖 —— 提示用户确认可信度。

#### 调试残留
- **调试打印（必须标记通知）：** 非测试代码中的 `console.log` / `console.debug` / `console.warn` / `print` / `printf` / `echo` / `var_dump` / `dd` / `dump` / `dbg!` / `println!` / `eprintln!` / `fmt.Println` / `log.Println` 等任何仅用于开发调试的无用输出语句。**发现时必须作为 🟡 警告通知标记，不可遗漏。**
- 无解释的注释代码块。
- 本次 diff 新增的 `TODO` / `FIXME`（提示，不必阻塞）。
- 临时值：`localhost`、`127.0.0.1`、假邮箱、`test123`。
- `.only` / `.skip` / `fdescribe` / `xit` 等测试标记。
- `debugger` 语句。

#### 项目约定（依据 AGENTS.md）
- 文件位置错误。
- 分层违规（如 nkc 中路由层直接访问 `ctx.db` 而未走 `services`）。
- 在要求 `.ts` 的项目中新增 `.js`。
- 使用被禁止的导出方式（如禁用 `export = xxx`）。
- 违反模型约定（如 Mongoose 上禁用 `method`/`static`）。
- 缺少路径别名（`../../../` 而非 `@/`）。
- 新 Vue 文件未使用要求的 `setup + ts + pug`。
- 新增路由未同步更新 `operations.ts` / `operationsTree`。

#### 代码质量
- 死代码、未使用的 import/变量、不可达分支。
- 重复逻辑 —— 建议复用已有实现。
- 过于复杂的函数。
- 缺少命名常量的魔法数/字符串。
- 命名与周围文件风格不一致。
- 公共 API 改动未同步调用方（快速 `grep` 旧符号）。

#### 类型 & API（静态类型语言）
- 无理由引入 `any` / `unknown`。
- 新增 `@ts-ignore` / `@ts-expect-error` / `eslint-disable` 必须有说明。
- 用类型断言 `as Foo` 掩盖真实的类型不匹配。
- 公共函数签名改动未同步 `types/` 定义。

#### 测试
- 项目有测试套件时，新行为却无测试。
- 测试没有实质断言。
- 快照更新疑似掩盖真实回归。

#### 文档 & i18n
- 公共 API / 配置 / CLI 选项变更未更新文档。
- 用户可见文案硬编码，未走 i18n（如 nkc 的 `languages/`）。

#### diff 卫生
- 无关改动混入 —— 建议拆分（见步骤 5）。
- 大段格式化混入逻辑改动 —— 建议分开提交。
- 大型生成文件（lockfile、构建产物）—— 确认是否应入库。
- 不应追踪的文件：`.DS_Store`、`Thumbs.db`、IDE 配置、`node_modules`、`dist`、`.env*`。
- 仅行尾/空白差异的噪声。
- 遗留的冲突标记：`<<<<<<<`、`=======`、`>>>>>>>`。

### 4. 交叉验证

改动可疑时**先核实再报告**：
- 函数签名变更 → `grep` 调用方。
- 文件删除/重命名 → `grep` 旧路径。
- 配置键重命名 → 搜索旧键。

### 5. 制定提交策略

审查完成后，按**模块或功能**拆分提交步骤。**始终分多步提交**，不存在单步提交选项。

**拆分原则：**
- 按模块/领域拆分（如 forum 相关改动一步、user 相关改动另一步）。
- 按功能主题拆分（如功能新增一步、bug 修复一步、依赖升级一步）。
- 大段格式化/重排与逻辑改动分开。
- 新文件入库与无关杂项改动分开。
- 若所有改动确实只属于单一模块的单一主题，则作为一步提交（仍属于分步提交中的特例，步骤数为 1）。

**拆分要求：**
- 每步可独立提交、独立回滚，不依赖其他步骤的中间状态。
- 每步文件清单明确，步骤之间文件不重叠。
- 步骤顺序符合依赖关系（如先底层后上层、先重构后功能）。

### 6. 输出审查结果与提交方案

按以下结构组织最终消息，全文使用**中文**（文件路径、行号、代码片段、命令、英文术语如 `console.log`、`await` 保持原样）。

```
## 提交前审查

**范围：** 已暂存 N 个文件，未暂存 M 个文件

### 🔴 阻塞问题
- `path/to/file.ts:42` —— <问题及原因>
  建议修复：<方案>

### 🟡 警告
- `path/to/file.ts:88` —— <需要再确认的点>

### 🟢 提示
- ...

### 总结
<一句话："可以提交" 或 "请先修复阻塞问题后再次审查">

### 提交方案
<格式见下>
```

#### 提交方案格式

```
**步骤 1 / N** —— <模块/主题说明>
信息：`<type>: <描述>`
文件：`path/a.ts`, `path/b.ts`

**步骤 2 / N** —— <模块/主题说明>
信息：`<type>: <描述>`
文件：`path/c.ts`
```

> 当所有改动仅属于单一模块的单一主题时，N=1，仍按上述格式输出。

**输出规则：**
- 始终使用 diff 新版本侧的行号。
- 无阻塞问题时总结为"可以提交"，并在方案后通过 `ask-user-choice` 弹窗请求用户确认（见步骤 7）。
- 有阻塞问题时总结为"请先修复阻塞问题后再次审查"，**不输出提交方案，不请求确认**。
- 不写"整体看起来不错"这类填充语。
- 不建议与本次 diff 无关的重构。

### 7. 请求确认并执行 commit

#### 7.1 请求确认

仅当**无阻塞问题**时，在输出方案后，**必须调用 `ask-user-choice` skill** 以弹窗方式请求用户确认，不在聊天框等待手动输入。

**弹窗问题文本必须自包含完整信息**，让用户仅看弹窗即可理解审查结果和提交方案，无需翻看聊天记录。问题文本须包含：

1. **审查结果摘要**：阻塞问题、警告、提示的数量；若有警告或提示，逐条列出简要说明（每条一行简述）。
2. **提交方案详情**：总步骤数，以及每一步的完整 commit message 和文件清单。
3. **确认提示语**。

问题文本模板（根据实际情况填充）：

```
📋 提交前审查报告

【审查范围】已暂存 X 个文件，未暂存 Y 个文件

【审查结果】
🔴 阻塞问题：X 个
🟡 警告：X 个
  - <警告1 简述>
  - <警告2 简述>
🟢 提示：X 个
  - <提示1 简述>

【提交方案】共 N 步

步骤 1/N
  commit: <type>: <描述>
  文件: <file1>, <file2>

步骤 2/N
  commit: <type>: <描述>
  文件: <file3>

确认后将自动逐步执行全部 commit。是否确认？
```

> - 无警告或提示时，对应行显示"0 个"且不列明细。
> - 阻塞问题为 0 时才弹窗（有阻塞问题不进入此流程）。
> - 构造 JSON 时，问题文本中的换行需转为 `\n`，文本内的双引号需转义为 `\"`。

调用方式（将模板填充后作为 question 值，注意 JSON 转义）：

```bash
py ~/.agents/skills/ask-user-choice/ask_choice.py --questions '[{"question":"<填充后的完整文本>","options":["确认提交","取消"]}]' --agent-label "提交前审查确认"
```

> `timeout_ms` 设为 `600000`。

**弹窗结果处理：**
- 用户选择"确认提交" → 进入 7.2 自动执行全部步骤。
- 用户选择"取消"或弹窗返回 `{"status":"cancelled"}` → 不执行任何 `git add` / `git commit`，告知用户已取消并等待指示。
- 用户选"其他"并输入自定义内容 → 按用户意图处理（如用户要求修改方案，则调整后重新弹窗确认）。

**不询问用户单步还是分步提交** —— 提交策略已在步骤 5 自动确定，用户只需确认或取消。

#### 7.2 执行 commit

收到用户确认后，**自动逐步执行全部 commit，无需人工干预**：

1. 对每一步（按方案中的步骤顺序）：
   - 仅暂存该步文件：`git add -- <file1> <file2> ...`
   - 提交：`git commit -m "<type>: <描述>"`
2. 全部步骤完成后，运行 `git --no-pager log -n <步数> --stat` 汇总结果并反馈。

**执行约束：**
- 严格按步骤对应的文件清单暂存，不夹带其他改动。
- commit 信息与方案中完全一致，不加 body、不加额外引号。
- 逐步自动执行，无需在每步之间再次请求用户确认。
- 执行过程中若发现某步文件存在阻塞问题（如 `git add` 后复查发现遗漏），停止该步及后续步骤并提示用户。
- 用户未确认或取消时，**不执行任何** `git add` / `git commit`。

## commit message 规则

- 格式：`<type>: <描述>`，不带 body。
- 整行控制在 50 字符左右。
- type 从以下候选按主要改动选择：
  - `feat`：新增功能 / 接口 / 页面。
  - `fix`：修复 bug / 错误逻辑。
  - `perf`：性能优化（不改变外部行为）。
  - `ref`：重构（不新增功能、不修 bug）。
  - `chore`：杂项（构建、依赖、脚本等）。
- 描述以动词开头（新增/修复/重构/优化/移除/调整），不以标点结尾。
- 示例：`feat: 新增专栏置顶接口及权限校验` / `fix: 修复登录后 Token 未刷新` / `perf: 优化帖子列表查询索引` / `ref: 抽离帖子服务层逻辑` / `chore: 升级 mongoose 到 8.5`。

## 不适用场景

- 工作区干净 —— 告知无内容可审查。
- 用户想要整个项目的架构审查 —— 属于另一类任务。
