# 提交前代码审查

在改动进入 git 历史前捕获问题。

## 工作流

### 1. 收集 diff

```sh
git --no-pager status
git --no-pager diff --stat
git --no-pager diff --staged
git --no-pager diff
```

若同时存在暂存和未暂存改动，默认只审查**暂存**部分；用户明确要求时再审查两者并分开报告。大 diff 可先 `git --no-pager diff --staged --name-only` 逐文件审查。

### 2. 加载项目约定

- 读取根目录及改动目录下的 `AGENTS.md`。
- 必要时读取 `README.md`。
- 项目规则优先级高于下方通用检查项。

### 3. 审查清单

只报告真实存在的问题，不要为了显得全面而编造。

#### 正确性 & bug
- 逻辑错误、off-by-one、条件反转、运算符错误。
- null/undefined 访问、遗漏 `await`、未处理的 promise rejection。
- 变量误用（能编译但行为错）。
- 资源泄漏：未关闭的文件、连接、监听器、定时器。
- 竞态与执行顺序问题。
- 错误处理：吞异常、空 `catch`、未向上传递错误。
- 边界情况：空数组、0、负数、超大输入、unicode。

#### 安全
- 硬编码的密钥、token、密码、内部 URL。
- 误提交的 `.env`、凭据、配置文件。
- SQL/NoSQL 注入、命令注入、路径穿越。
- 未转义的用户输入渲染到 HTML（XSS）或拼接到 shell。
- 权限校验被削弱或移除、CORS 放宽。
- 新增外部依赖 —— 提示用户确认可信度。

#### 调试残留
- 非测试代码中的 `console.log` / `print` / `dbg!` / `println!` 等。
- 无解释的注释代码块。
- 本次 diff 新增的 `TODO` / `FIXME`（提示，不必阻塞）。
- 临时值：`localhost`、`127.0.0.1`、假邮箱、`test123`。
- `.only` / `.skip` / `fdescribe` / `xit` 等测试标记。
- `debugger` 语句。

#### 项目约定（依据 AGENTS.md）
- 文件位置错误。
- 分层违规（如 nkc 中路由层直接访问 `ctx.db` 而未走 `services`）。
- 在要求 `.ts` 的项目中新增 `.js`。
- 使用被禁止的导出方式（如禁用 `export = xxx`）。
- 违反模型约定（如 Mongoose 上禁用 `method`/`static`）。
- 缺少路径别名（`../../../` 而非 `@/`）。
- 新 Vue 文件未使用要求的 `setup + ts + pug`。
- 新增路由未同步更新 `operations.ts` / `operationsTree`。

#### 代码质量
- 死代码、未使用的 import/变量、不可达分支。
- 重复逻辑 —— 建议复用已有实现。
- 过于复杂的函数。
- 缺少命名常量的魔法数/字符串。
- 命名与周围文件风格不一致。
- 公共 API 改动未同步调用方（快速 `grep` 旧符号）。

#### 类型 & API（静态类型语言）
- 无理由引入 `any` / `unknown`。
- 新增 `@ts-ignore` / `@ts-expect-error` / `eslint-disable` 必须有说明。
- 用类型断言 `as Foo` 掩盖真实的类型不匹配。
- 公共函数签名改动未同步 `types/` 定义。

#### 测试
- 项目有测试套件时，新行为却无测试。
- 测试没有实质断言。
- 快照更新疑似掩盖真实回归。

#### 文档 & i18n
- 公共 API / 配置 / CLI 选项变更未更新文档。
- 用户可见文案硬编码，未走 i18n（如 nkc 的 `languages/`）。

#### diff 卫生
- 无关改动混入 —— 建议拆分。
- 大段格式化混入逻辑改动 —— 建议分开提交。
- 大型生成文件（lockfile、构建产物）—— 确认是否应入库。
- 不应追踪的文件：`.DS_Store`、`Thumbs.db`、IDE 配置、`node_modules`、`dist`、`.env*`。
- 仅行尾/空白差异的噪声。
- 遗留的冲突标记：`<<<<<<<`、`=======`、`>>>>>>>`。

### 4. 交叉验证

改动可疑时**先核实再报告**：
- 函数签名变更 → `grep` 调用方。
- 文件删除/重命名 → `grep` 旧路径。
- 配置键重命名 → 搜索旧键。

### 5. 输出报告

按以下结构组织最终消息，全文使用**中文**（文件路径、行号、代码片段、命令、英文术语如 `console.log`、`await` 保持原样）。

```
## 提交前审查

**范围：** 已暂存 N 个文件，未暂存 M 个文件

### 🔴 阻塞问题
- `path/to/file.ts:42` —— <问题及原因>
  建议修复：<方案>

### 🟡 警告
- `path/to/file.ts:88` —— <需要再确认的点>

### 🟢 提示
- ...

### 总结
<一句话："可以提交" 或 "请先修复阻塞问题后再次审查">

### 推荐提交信息
```
<type>: <简短中文描述>
```
```

**规则：**
- 始终使用 diff 新版本侧的行号。
- 无问题时明确放行。
- 不写"整体看起来不错"这类填充语。
- 不建议与本次 diff 无关的重构。
- **不要**执行 `git commit`、`git add`、`git reset` 等改动仓库状态的命令；也不要修改文件，除非用户明确要求应用某条修复。

**推荐提交信息规则：**
- 直接输出 `<type>: <描述>`，不含 `git commit -m`，不加引号，不带 body。
- 整行控制在 50 字符左右。
- type 从以下候选按主要改动选择：
  - `feat`：新增功能 / 接口 / 页面。
  - `fix`：修复 bug / 错误逻辑。
  - `perf`：性能优化（不改变外部行为）。
  - `ref`：重构（不新增功能、不修 bug）。
  - `chore`：杂项（构建、依赖、脚本等）。
- 涵盖多类型时选影响最大的，并在警告区提示考虑拆分。
- 描述以动词开头（新增/修复/重构/优化/移除/调整），不以标点结尾。
- 存在阻塞问题时仍给出推荐信息，但在总结中提醒"修复后再使用"。
- 示例：`feat: 新增专栏置顶接口及权限校验` / `fix: 修复登录后 Token 未刷新` / `perf: 优化帖子列表查询索引` / `ref: 抽离帖子服务层逻辑` / `chore: 升级 mongoose 到 8.5`。

## 不适用场景

- 工作区干净 —— 告知无内容可审查。
- 用户想要整个项目的架构审查 —— 属于另一类任务。
- 用户想要生成完整 commit message —— 由用户的个人 commit-message 指令负责。
