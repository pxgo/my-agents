# 代码修改强制规则

> 核心：不确定先问、等回复确认理解后再动手；复杂任务小步推进、每步校验、禁止越权重构；简单任务直接处理。

---

## 表达原则：精简省 token

保留必要的逻辑推理与分析（确认理解、影响判断、根因定位、自检推理不可省），但砍掉一切无信息量的口水话：禁止复述用户原话、禁止重复已得出的结论、禁止套话与铺垫、禁止"我将要…"式过程叙述。同一信息只说一次，后续仅引用，不再展开。目的是节约 token、减小上下文。

---

## 前置门禁：不清楚先确认（最高优先级）

需求、上下文、影响范围、多种合理实现之间有任何不确定 -> **先问用户并等待回复**，禁止"边问边改"。收到回复后复述理解请用户确认，仍有歧义则继续追问，确认无误后方可动手。

- 即便看似"显而易见"，只要存在多种合理实现或影响范围不明，都必须先问。
- 有 2 个及以上离散选项时，**必须**用 `ask-user-choice` skill 弹窗提问（问题须自包含上下文）；开放式问题直接在对话中问。

---

## 任务分级

先判断**简单**还是**复杂**，再走对应流程。

### 简单任务 -- 直接处理
满足任一：单文件单一处小改动 / 无接口类型行为变更 / 无跨模块影响 / 显而易见的修补（拼写、常量、局部日志、注释）。
-> 直接改，无需拆步骤、无需逐步校验，改完确认无误即可。

### 复杂任务 -- 严格分步
满足任一：多文件或单文件多处改动 / 涉及接口类型对外行为变更 / 有跨模块影响 / 逻辑不直观存在多种实现或较大风险。

---

## 复杂任务主流程

1. **分析与拆分**：理清依赖、数据流、影响范围；拆成最小原子步骤，每步说明改什么/改哪/位置/风险。禁止无计划直接改、多个改动混在一步。
2. **最小化修改**：只改必要代码，保持函数签名、类型、对外接口、模块结构、命名风格；禁止无关重构、重命名、格式化、架构调整。代码改了相关注释须同步更新。
3. **修改后自检**：每步落笔后查语法/类型/边界（空值、极端）/影响（是否破坏现有逻辑），发现问题立即修。
4. **闭环确认**：进入下一步前确保无已知缺陷、检查全过、一致性完好、逻辑闭环。

---

## 子代理分派策略（仅复杂任务考虑）

遇到复杂任务时**必须考虑**用 `spawn_agent` 委派，但无必要不拆分。

- **单个子代理**：需读大量文件理解全貌、深度调研、跨模块追踪、复杂缺陷定位，或单次任务会吃掉大量上下文 -> 子代理只回传结论与关键片段。
- **并行多个子代理**：改动集可拆成**写入范围完全不重叠**且**无先后依赖**的独立子任务 -> 主代理只做派发、汇总、一致性复核。
- **分派要求**：每个子任务自包含（目标/涉及文件/约束/验收标准/必须遵循本 skill）；写入范围严格隔离；子代理完成后主代理必须复核是否引入回归、风格是否一致。
- **禁止**：把有依赖的步骤强行并行；派完不复核。

好例：三个独立页面各加埋点 -> 并行三个子代理；定位跨十几个文件的缺陷 -> 单派一个调研子代理只回传根因。
反例：改一个函数三行代码别派子代理；先改 schema 再改服务层有依赖必须串行；派完不复核属违规。

---

## 适用范围

所有代码写入操作（单行/多文件、新增/修复/重构/优化、接口/类型/配置/依赖/数据结构/前端组件/样式/模板等）。纯只读操作（查询、解释、答疑）可跳过。
