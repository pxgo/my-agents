# 代码修改强制规则

> 核心：不确定先问且必须等回复确认理解后才能动手、小步推进、每步校验、禁止越权重构。

---

## 主流程

### 步骤 0 — 不清楚先确认（前置门禁，最高优先级）

需求、上下文、影响范围、多种合理实现之间有任何不确定 → **先问用户**，并严格遵守以下硬顺序：

1. **提问**：列出不确定点，给出候选方案或默认建议，请用户拍板。
2. **等待回复**：提问后必须停下，禁止"边问边改"或继续推进任何修改动作。
3. **确认理解**：收到回复后复述自己的理解请用户确认；若仍有歧义，回到第 1 步继续追问，禁止"问完就猜着改"。
4. **才能动手**：用户明确确认且自己理解清楚后，方可进入步骤 1。

**门禁约束**：未获得用户明确确认前，**禁止进入步骤 1（分析与拆分）**，更禁止直接落笔修改。即便看似"显而易见"，只要存在多种合理实现或影响范围不明，都必须先问。

#### 提问方式：有离散选项时必须使用弹窗

当问题有 **2 个及以上离散选项**时，禁止在聊天框文字罗列选项等待用户手动输入，必须读取并遵照 `ask-user-choice` skill 的说明通过弹窗提问。

问题描述必须自包含足够上下文，确保用户仅凭弹窗即可理解并作答，无需再查看聊天记录。

开放式问题（无固定选项）仍在对话中直接提问。

---

### 步骤 1 — 分析与拆分
- 先理清依赖、数据流、影响范围
- 拆成最小原子步骤，每步说明：改什么 / 改哪 / 位置 / 风险
- 禁止：无计划直接改、多个改动混在一步

### 步骤 2 — 最小化修改
- 只改必要代码
- 保持：函数签名、类型、对外接口、模块结构、命名风格
- 禁止：无关重构、重命名、格式化、架构调整

### 步骤 3 — 修改后立即自检
每步落笔后必查：语法 / 类型 / 边界（空值、极端）/ 影响（是否破坏现有逻辑）。
发现问题立即修，不带问题进下一步。

### 步骤 4 — 闭环确认
进入下一步前：无已知缺陷、检查全过、一致性完好、逻辑闭环。
若当前问题需靠后续步骤才能修 → 调整顺序，优先修当前。

---

## 轻量模式（简单改动可合并流程）

**同时满足下列全部条件时**，可合并步骤 1-4 为一次输出：

- 单文件、单一处改动，且行数极少
- 无接口、类型、行为语义变更
- 无跨模块影响（不涉及导出符号、共享状态、配置）
- 属于显而易见的修补（拼写、常量值、局部日志、注释）

即便走轻量模式，**自检项（语法 / 类型 / 边界 / 影响）不可省**，并额外自检"是否真的显而易见、是否只有一种合理做法"——若有任何疑虑，回到步骤 0 先问。
任一条件不满足 → 回到完整主流程。

## 子代理分派策略

遇到下列情况**必须考虑**用 `spawn_agent` 委派。

### 何时启动**单个**子代理
- 需读大量文件才能理解全貌
- 深度调研、跨模块追踪、复杂缺陷定位
- 单次任务预计吃掉大量上下文长度，会污染主线

→ 子代理只回传结论与关键片段，主线保持清洁。

### 何时**并行**启动多个子代理
- 改动集可拆成**写入范围完全不重叠**的独立子任务
- 各子任务之间**无先后依赖**
- 例：模块 A、模块 B、配置 C 同时需要独立改动

→ 主代理只做派发、汇总、一致性复核。

### 分派要求
- 每个子任务自包含：目标 / 涉及文件 / 约束 / 验收标准 / 必须遵循本 skill
- 写入范围严格隔离，避免文件冲突
- 子代理完成后主代理必须复核：是否遵循本 skill、是否引入回归、风格是否一致

### 好例 vs 反例

**好例（应并行）**
> 任务：给三个独立页面各加一个埋点。
> 派发：三个子代理，各负责一个页面文件；主代理汇总后统一自检。

**好例（应单派）**
> 任务：定位一个跨十几个文件的复杂缺陷。
> 派发：一个子代理专职调研，只回传"根因 + 建议改法"；主代理据此在主线执行修改。

**反例（不应拆）**
> 任务：改一个函数的三行代码。
> → 直接改，别派子代理，派发成本高于收益。

**反例（不应并行）**
> 任务：先改数据模型 schema，再改用它的服务层。
> → 有依赖，必须串行，不能同时派发。

**反例（不做复核）**
> 子代理完成后直接采纳输出，未验证是否引入回归。
> → 违规，主代理必须复核。

### 禁止
- 无必要就拆分
- 把有依赖的步骤强行并行化
- 派完不复核

---

## 适用范围

所有代码写入操作：单行 / 多文件、新增 / 修复 / 重构 / 优化、接口 / 类型 / 配置 / 依赖 / 数据结构 / 前端组件 / 样式 / 模板 等。

纯只读操作（查询、解释、答疑）可跳过本规则。

---

## 行为准则

像资深工程师一样：**正确性优先于速度，小步可验证推进；不清楚先问、等回复确认理解后再动手。**
